# Information Architecture

## Design rationale

### Diátaxis slots chosen (and skipped)

| Slot | Purpose | Used here? | Why / why not |
|---|---|:-:|---|
| **Tutorial** | Learning-oriented, guided first-time experience | ❌ | Audience (internal testers + team) already knows Gen1POS; tutorial-level onboarding would duplicate parent manual. |
| **How-to** | Task-oriented, step-by-step for a specific goal | ✅ | Primary format for the 5 happy-path scenarios from `07-scenarios.md`. |
| **Reference** | Information-oriented, look-up content | ✅ | Rules, glossary, limitations — the "I'm in the middle of testing, what's the exact behaviour?" use case. |
| **Explanation** | Understanding-oriented, why and what | ✅ (compact) | One short concept page to give the mental model. Audience doesn't need long theory — they need an anchor. |

### Other design choices

- **Link-out over duplicate** for base concepts (item, modifier, category, etc.). They live in the parent Gen1POS manual per PO Round 1 Q-1.
- **Short, focused files** over one long document. A tester looking up "what happens on detach" shouldn't scroll through setup docs.
- **Reference is structured as tables**, not prose. Faster lookup, explicit structure.
- **`08-known-limitations.md` is a first-class document**, not an appendix. Per ISO/IEC/IEEE 26514 safe-scope publication principle, what is NOT documented must be explicit.

## File-by-file specification

### 00-index.md (Meta / Navigation)

- **Diátaxis slot**: n/a (meta).
- **Length target**: ≤ 1 screen (~100 lines).
- **Contents**:
  - 2–3 sentence positioning: "This section covers the Parent-Child Structure feature. It extends the base item configuration described in *[link to parent manual]*."
  - Table of contents with 1-line descriptions.
  - "Who this is for" (testers, internal team).
  - Version / last updated.
- **Sources**: `01-task-contract.md` (positioning), `03-audience.md` (who-for).

### 01-explanation.md (Explanation)

- **Diátaxis slot**: Explanation — understanding-oriented.
- **Length target**: ≤ 1 page (~80–120 lines). Compact by design.
- **Contents**:
  - What Parent-Child Structure is (one-paragraph definition).
  - The core mental model: Parent item + Variants (children), linked by `Parent Item` field.
  - Hierarchy diagram (Mermaid or static image): Category → Subcategory → Item (Parent) → Variants. Based on `plan.md` section 1.3.
  - Two ways a variant comes into existence (create new vs attach existing) — 1–2 sentences each, no steps (those live in how-tos).
  - Where it shows up across the product: admin, POS, reports. One-line summary each, links to Reference.
  - No step-by-step instructions — those are how-tos.
  - No rule tables — those are Reference.
- **Sources**: CAP-1 through CAP-12 (summary form), `02-glossary.md`.

### 06-reference-rules.md (Reference — Rules & Behaviour)

- **Diátaxis slot**: Reference.
- **Length target**: 150–250 lines.
- **Sections** (each a table or concise rule block):
  - **R1. Where Parent-Child appears** — surface matrix (Admin | POS | Kitchen | ProductMix) × what is shown. Based on CAP-8, CAP-9, CAP-10, CAP-11.
  - **R2. Short Name** — when it appears, where it's used. CAP-3.
  - **R3. Product Class inheritance** — single rule statement + example. CAP-7 (NN-only; phrasing must be careful per Source Matrix).
  - **R4. Subcategory scope constraint** — CAP-12.
  - **R5. Detach behaviour** — branch table (standalone-origin vs variant-born). CAP-5.
  - **R6. Display rules**:
    - Admin — Variants section list format.
    - POS — label = Short Name, order = saved order, pre-chosen visual.
    - Order preview — two-line "parent name + short name" format.
    - Kitchen ticket — **phrase carefully** (C14 / UK7 — format not verifiable).
    - ProductMix — composite label observed as `Parent Name – Short Name Parent Name` (C15 / resolution of F10-vs-F12 contradiction).
  - **R7. Modifiers on variants** — structural only (fields exist); behavioural note: "Exact interaction with parent-child context is not documented in v1 — see Known Limitations." (C11 / UK10).
  - **R8. Color code** — display-option field. Behavioural caveat same as R7 (C16 / UK10).
- **Sources**: CAP-3, CAP-6, CAP-7, CAP-8, CAP-9, CAP-10, CAP-11, CAP-12.
